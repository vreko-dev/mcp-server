# Contributing to SnapBack

We welcome contributions from the community! This guide covers how to get started with SnapBack development, the package structure, and contribution guidelines.

## Getting Started

### Prerequisites

- **Node.js** 20+ ([Download](https://nodejs.org))
- **pnpm** 9+ (`npm install -g pnpm`)
- **Git**

### Setting Up Your Development Environment

```bash
# 1. Clone the repository
git clone https://github.com/snapback-dev/snapback.dev.git
cd snapback-dev

# 2. Install dependencies
pnpm install

# 3. Build all packages
pnpm build

# 4. Run tests
pnpm test

# 5. Start development
pnpm dev
```

## Monorepo Structure

SnapBack uses a monorepo managed by [pnpm workspaces](https://pnpm.io/workspaces) and [Turborepo](https://turbo.build).

```
snapback/
├── apps/                      # Applications
│   ├── docs/                  # Documentation site
│   ├── web/                   # Main web application
│   ├── vscode/                # VS Code extension
│   ├── mcp-server/            # MCP server implementation
│   └── cli/                   # Command-line tool
├── packages/                  # Private packages
│   ├── contracts/             # Type definitions (public & private)
│   ├── core/                  # Core business logic
│   ├── infrastructure/        # Logging, metrics, tracing
│   ├── sdk/                   # SDK with full features
│   └── events/                # Event bus
├── packages-oss/              # Public open-source packages
│   ├── contracts/             # Public type definitions
│   ├── infrastructure/        # Public utilities
│   ├── sdk/                   # Public SDK (limited features)
│   ├── config/                # Configuration utilities
│   └── events/                # Event bus
├── pnpm-workspace.yaml        # Workspace configuration
├── turbo.json                 # Build orchestration
└── tsconfig.base.json         # TypeScript configuration
```

### Package Naming Convention

- **Private packages**: `@snapback/*` (e.g., `@snapback/sdk`, `@snapback/core`)
- **Public packages**: `@snapback-oss/*` (e.g., `@snapback-oss/sdk`, `@snapback-oss/contracts`)

## Key Packages

### Core Business Logic

**`@snapback/core`**
- Core business logic for snapshot management
- File protection and restoration
- Risk assessment and analytics
- **Access**: Private only

**`@snapback/sdk`**
- Full-featured SDK with all capabilities
- Storage adapters and integrations
- **Access**: Private (used by apps), published to npm as `@snapback/sdk`

### Public/Open-Source

**`@snapback-oss/sdk`**
- Limited public SDK for OSS users
- Privacy-first design
- Safe for public consumption
- Published to npm

**`@snapback-oss/contracts`**
- Public type definitions
- Generated OpenAPI specs
- Event schemas
- Published to npm

**`@snapback-oss/infrastructure`**
- Generic logging and metrics
- No proprietary analytics
- Published to npm

### Shared Utilities

**`@snapback/contracts`**
- Master type definitions
- Both public and private exports
- Events schemas
- Generated API documentation

**`@snapback/infrastructure`**
- Infrastructure utilities
- Logging (Pino)
- PostHog integration (private)
- Metrics and tracing

**`@snapback/events`**
- Event bus implementation
- Pub/sub patterns

## Development Workflow

### 1. Creating a Feature

```bash
# Create feature branch
git checkout -b feature/my-feature

# Make your changes
# Follow the code style guidelines below

# Run tests
pnpm test

# Type check
pnpm type-check

# Lint
pnpm lint
```

### 2. Package-Specific Development

```bash
# Work on a specific package
pnpm -F @snapback/sdk dev

# Run tests for a package
pnpm -F @snapback/contracts test

# Build a package
pnpm -F @snapback/sdk build
```

### 3. Committing Changes

We use conventional commits:

```bash
git commit -m "feat(sdk): add snapshot validation"
git commit -m "fix(infrastructure): improve logger performance"
git commit -m "docs(contracts): update type definitions"
```

Commit types: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`

### 4. Submitting a Pull Request

1. **Push your branch** to your fork
2. **Create a PR** on GitHub with a clear description
3. **Ensure CI passes**: Tests, linting, type checking
4. **Address review feedback**
5. **Maintainers will merge**

## Code Style Guidelines

### TypeScript

We follow strict TypeScript patterns:

- **Discriminated Unions**: Use for state machines and result types
- **Type Guards**: Use `is` predicates for type narrowing
- **Const Assertions**: Use `as const` for readonly data
- **Result Types**: Use `Result<T, E>` pattern for error handling

**Example:**
```typescript
// ✅ Discriminated union for state
type SnapshotState =
  | { status: "loading" }
  | { status: "ready"; data: Snapshot }
  | { status: "error"; error: Error };

// ✅ Type guard
function isReady(state: SnapshotState): state is { status: "ready"; data: Snapshot } {
  return state.status === "ready";
}

// ✅ Result type for error handling
async function createSnapshot(path: string): Promise<Result<Snapshot, Error>> {
  try {
    const snapshot = await storage.create(path);
    return Ok(snapshot);
  } catch (err) {
    return Err(toError(err));
  }
}
```

### Imports

Use package imports for cross-package dependencies:

```typescript
// ✅ CORRECT - Cross-package imports use @snapback/* namespace
import { logger } from "@snapback/infrastructure";
import type { Snapshot } from "@snapback/contracts";

// ❌ WRONG - Don't use relative paths across packages
import { logger } from "../../../packages/infrastructure/src/logging";
```

### Comments

- **English only** for code comments
- **Chinese allowed** for business logic explanations
- **Document public APIs** with JSDoc
- **Mark TODOs** with issue numbers: `// TODO: #123`

## Contributing to OSS Packages

Only the following packages are public open-source:

- `@snapback-oss/contracts`
- `@snapback-oss/sdk`
- `@snapback-oss/infrastructure`
- `@snapback-oss/config`
- `@snapback-oss/events`

### Rules for OSS Packages

**Never add**:
- Database schemas or migrations
- Proprietary algorithms
- Analytics or telemetry code (use `@snapback-oss/infrastructure` for logging only)
- Subscription/tier logic
- Internal platform code
- Hardcoded API keys or secrets

**Always ensure**:
- No imports from private packages (`@snapback/core`, `@snapback-oss/*` cannot import `@snapback/*`)
- Dependencies are safe for public consumption
- Types don't expose private implementation details
- Tests pass with public dependencies only

### Testing OSS Packages

```bash
# Test all OSS packages
pnpm --filter "@snapback-oss/*" test

# Build all OSS packages
pnpm --filter "@snapback-oss/*" build

# Type check
pnpm --filter "@snapback-oss/*" type-check
```

## Testing

### Writing Tests

Tests use [Vitest](https://vitest.dev) for unit tests and integration tests.

```typescript
import { describe, it, expect } from "vitest";
import { createSnapshot } from "@snapback/sdk";

describe("Snapshot Creation", () => {
  it("should create a snapshot successfully", async () => {
    const result = await createSnapshot("/path/to/file.ts");

    expect(isOk(result)).toBe(true);
    if (isOk(result)) {
      expect(result.value).toHaveProperty("id");
      expect(result.value).toHaveProperty("timestamp");
    }
  });

  it("should return error for invalid path", async () => {
    const result = await createSnapshot("");

    expect(isErr(result)).toBe(true);
    if (isErr(result)) {
      expect(result.error).toBeInstanceOf(Error);
    }
  });
});
```

### Running Tests

```bash
# Run all tests
pnpm test

# Run tests in watch mode
pnpm test --watch

# Run tests for a specific package
pnpm -F @snapback/sdk test

# Run tests with coverage
pnpm test --coverage
```

## Reporting Issues

Found a bug? We appreciate detailed bug reports:

1. **Search existing issues** to avoid duplicates
2. **Provide a minimal reproduction** (code snippet or test case)
3. **Include environment** (Node.js version, OS, etc.)
4. **Describe expected vs actual behavior**
