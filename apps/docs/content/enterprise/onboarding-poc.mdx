# Enterprise POC Framework

A structured 2-week proof-of-concept program to evaluate SnapBack with quantitative success criteria.

## POC Timeline

### Week 1: Setup & Integration

**Days 1-2: Environment Setup**
- Install SnapBack extension for pilot team (5-10 developers)
- Configure SSO/SAML integration (if applicable)
- Set up cloud backup and Guardian detection
- Configure team-wide .snapbackrc policies

**Days 3-4: Baseline Measurement**
- Begin snapshot capture on production work
- Establish baseline metrics:
  - Average snapshot creation latency
  - Guardian detection false positive rate
  - Developer workflow friction points
- Initial training session with pilot team

**Day 5: First Check-in**
- Review baseline metrics with CSM
- Address any technical issues
- Adjust policies based on feedback
- Plan Week 2 production pilot

### Week 2: Production Pilot & Validation

**Days 6-8: Production Workloads**
- Pilot team uses SnapBack on real feature work
- Monitor Guardian detection accuracy
- Test session rollback on actual AI-assisted changes
- Collect developer satisfaction feedback

**Days 9-10: Success Validation**
- Generate Recover Report with quantitative results
- Evaluate against success criteria (see below)
- Conduct team retrospective
- Decision point: proceed to deployment or adjust

## Success Criteria Template

Define objective metrics before starting POC. Recommended criteria:

### Performance Metrics

| Metric | Target | Pass/Fail |
|--------|--------|-----------|
| Snapshot creation latency | < 200ms | Required |
| Guardian detection accuracy | > 90% | Required |
| False positive rate | < 5% | Required |
| Zero critical file blocks (false) | 0 occurrences | Required |

### Adoption Metrics

| Metric | Target | Pass/Fail |
|--------|--------|-----------|
| Team adoption rate | > 80% daily active | Recommended |
| Snapshots per developer per day | > 10 | Recommended |
| Session rollbacks executed | > 3 total | Recommended |

### Value Metrics

| Metric | Target | Pass/Fail |
|--------|--------|-----------|
| Prevented secret commits | > 2 caught | Success indicator |
| Time saved via rollback | > 2 hours total | Success indicator |
| Developer satisfaction (1-5) | ≥ 4.0 average | Success indicator |

## Rollback Plan

If POC doesn't meet success criteria, safe exit strategy:

### Data Retention

- **Local snapshots**: Remain on developer machines (no cleanup needed)
- **Cloud backups**: Retained for 30 days after POC end
- **Team data**: Export all data before POC cancellation

### Cleanup Steps

1. **Disable SSO** (if configured): Remove SnapBack from IdP
2. **Export data**: Download Recover Report and all snapshot metadata
3. **Uninstall extension**: Optional; developers can keep for local use
4. **Account closure**: Close Enterprise trial account

### No-Risk Guarantee

- No data loss during or after POC
- No credit card required for POC period
- Full data export before account closure
- Local snapshots never affected by POC cancellation

## Evaluation Scorecard

Use this scorecard to make your deployment decision:

### Technical Performance (40 points)

- [ ] Snapshot latency < 200ms (10 points)
- [ ] Guardian accuracy > 90% (10 points)
- [ ] False positives < 5% (10 points)
- [ ] Zero critical failures (10 points)

### Team Adoption (30 points)

- [ ] >80% daily active users (15 points)
- [ ] >10 snapshots/developer/day (10 points)
- [ ] Positive feedback from majority (5 points)

### Business Value (30 points)

- [ ] Prevented ≥2 secret commits (10 points)
- [ ] Saved ≥2 hours via rollback (10 points)
- [ ] Developer satisfaction ≥4.0 (10 points)

**Scoring**:
- **90-100 points**: Strong deployment candidate
- **70-89 points**: Deploy with minor adjustments
- **50-69 points**: Extend POC or adjust policies
- **<50 points**: Exit POC or re-evaluate fit

## After POC: Next Steps

### If POC Succeeds

1. **Contract negotiation**: Finalize Enterprise agreement
2. **SSO production**: Configure SSO for full organization
3. **Rollout plan**: Phase deployment to all teams
4. **Success monitoring**: Continue metrics tracking

### If POC Needs Adjustment

1. **Gap analysis**: Identify specific issues
2. **Policy tuning**: Adjust Guardian rules, protection levels
3. **Extended POC**: Add 1-2 weeks to re-validate
4. **Re-evaluate**: Second scorecard assessment

### If POC Exits

1. **Exit interview**: Understand reasons for exit
2. **Data export**: Provide full data package
3. **Cleanup**: Disable SSO, close accounts
4. **Future re-evaluation**: Keep door open for later

## Frequently Asked Questions

### How long does POC take?

Standard POC is 2 weeks. We recommend this timeframe to capture enough real-world usage data. Extended POCs (3-4 weeks) available if needed.

### Can we extend the POC?

Yes! If you need more time to validate, we can extend 1-2 weeks at no cost.

### What if our team is larger than 10?

POCs typically use 5-10 pilot users. For larger teams, we can accommodate up to 25 pilots with adjusted success criteria.

### Do we need to configure SSO for POC?

No, SSO is optional during POC. You can evaluate with email/password logins and add SSO before full deployment.

### What data do we get to keep?

You get to keep everything: Recover Report, all snapshot metadata, Guardian detection logs, and of course all local snapshots remain on developer machines.

### Is there a minimum commitment after POC?

No. POC is risk-free with no obligation to purchase. If it doesn't meet your criteria, you can exit with full data export.
